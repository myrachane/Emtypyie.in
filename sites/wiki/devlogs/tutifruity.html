§1 Overview

TutiFruITy chAn is a slash-command-only Discord bot built on discord.py 2.7. Every reply she gives is pulled from a centralized personality system instead of being hardcoded per-command, so she can have moods without the command logic ever knowing or caring what she actually says.

Core features:

  • Centralized tsundere personality engine - dozens of randomized lines per situation, kaomoji included
  • Moderation suite: /ban /kick /timeout /untimeout /purge /warn
  • /modrole - delegate moderation access to a role without touching native Discord permissions
  • /sauce - MangaDex-backed manga lookup: random pulls, genre/theme filters with live autocomplete, cover art
  • /randomquote - a pool of mostly sarcastic, troll-tier one-liners
  • Music - currently disabled, mid-rebuild around the Spotify API (see dev log #2)

One shared SQLite connection handles warnings and mod-role config - no module ever opens its own.

§2 Why I built this

Most moderation bots feel the same - same dry embeds, same robotic "Banned." I wanted something that still does the job properly (bans happen, warnings get logged, channels get purged) but acts mildly inconvenienced about doing it the whole time.

It also turned into a decent crash course in discord.py's app_commands system - cogs, custom permission checks, autocomplete, error handling - in a way a toy project doesn't usually force you to learn.

§3 Architecture / Design

main.py creates the bot, opens the single SQLite connection, and loads every module as a cog. functions.py holds nothing but reusable helpers - the personality response system, permission checks, db helpers - no commands live there. moderation.py, utility.py, sauce.py, and music.py (currently unloaded) each hold one module's worth of slash commands and nothing else.

main.py
python 
class TsundereBot(commands.Bot):
    def __init__(self) -> None:
        intents = discord.Intents.default()
        intents.message_content = True
        intents.members = True
        intents.voice_states = True

        super().__init__(command_prefix="!", intents=intents)

        # Single, shared SQLite connection - never opened anywhere else.
        self.db: sqlite3.Connection = sqlite3.connect(config.DATABASE_PATH)
        self.db.row_factory = sqlite3.Row
        self._init_database()

    async def setup_hook(self) -> None:
        for extension in INITIAL_EXTENSIONS:
            try:
                await self.load_extension(extension)
                log.info("Loaded extension: %s", extension)
            except Exception:
                log.exception("Failed to load extension: %s", extension)

        synced = await self.tree.sync()
        log.info("Synced %d slash command(s).", len(synced))
functions.py
python 
def get_response(category: str, **kwargs) -> str:
    """
    Return a random tsundere line for the given category.
    Supports str.format kwargs (e.g. latency=42).
    Falls back to UNKNOWN_ERROR if the category doesn't exist.
    """
    pool = RESPONSES.get(category, RESPONSES["UNKNOWN_ERROR"])
    line = random.choice(pool)
    if kwargs:
        try:
            line = line.format(**kwargs)
        except (KeyError, IndexError):
            pass
    return line

v1.1.0
v1.1.0 — Sauce overhaul, /modrole, music paused for Spotify

Turns out /sauce random was broken from the start. MangaDex's /manga/random endpoint only returns a single entity and rejects unexpected query params like limit - it was silently 400ing on every call and quietly falling through to "not found." Rebuilt it around the documented /manga search endpoint instead: probe the total match count for the given filters, then re-query at a random offset into that count.

python 
async def _random_manga(self, tag_ids=None):
    probe = await self._get("/manga", [*base_params, ("limit", "1"), ("offset", "0")])
    total = probe.get("total", 0)
    offset = random.randint(0, min(total - 1, 9999))
    data = await self._get("/manga", [*base_params, ("limit", "1"), ("offset", str(offset))])
    return data["data"][0]

The same trick now powers /sauce category too - genre and theme filters with live autocomplete pulled straight from MangaDex's own tag list, plus cover art on every result.

Also shipped /modrole so a server can delegate the moderation commands to a role without granting real Discord permissions, added /randomquote, and sprinkled kaomoji through the personality lines. Music got disabled - it's being rebuilt around the Spotify API instead of YouTube, which is a bigger architectural change than a patch.

⚠ Known limitation: Spotify's API doesn't hand you raw audio to stream the way yt-dlp did - it's metadata plus remote control of an active Spotify Connect session on a Premium account. The music module needs a different architecture, not just a swapped extractor. Tracked for a future pass.

v1.0.0
Initial release — personality engine, moderation, sauce, music

First build. Got the full module structure up from spec: main.py wires the bot and the one shared SQLite connection, functions.py centralizes the tsundere RESPONSES dict behind a single get_response() call, and moderation/utility/music/sauce each became their own cog.

First boot hit two real-world snags immediately: discord.py now requires a separate davey package for Discord's mandatory E2EE voice protocol, and FFmpeg wasn't on PATH yet for music playback. Both fixed before /play ever successfully ran. YouTube also intermittently blocked yt-dlp with a "confirm you're not a bot" check - spoofing the Android/iOS client in the extractor args cleared most of it, with a browser-cookie fallback for the rest. Moot for now since music is disabled, but worth keeping around for the Spotify rebuild if YouTube ever comes back as a fallback source.
