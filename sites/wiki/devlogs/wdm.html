§1 Overview

W.Decision-Matrix is a fully client-side weighted decision matrix tool — one HTML file, no build pipeline, no npm install, no server. Open it in a browser and it works.

The idea is simple: you define criteria (cost, speed, quality — whatever matters) and give each a weight from 1–10 via a range slider. Then you add options (Vendor A, Vendor B, doing nothing). Score every option against every criterion with a stepper widget (0–10), and the results panel calculates a weighted score in real time, ranks everything, and animates the bars to show you the winner.

Core features:

  • Weighted criteria with live range sliders (1–10)
  • Score matrix with +/− steppers — no keyboard needed
  • Real-time ranked results with animated progress bars
  • Dark / light theme toggle with localStorage persistence
  • Reset button (with confirm guard so you don't nuke yourself)
  • Zero dependencies — ships as a single .html file
§2 Why I built this

Every time I had to make a decision with more than two options I'd either open a spreadsheet and spend 20 minutes setting up formulas, or just flip a coin. Neither is great. There are online tools for this but they're either bloated, behind a login, or ugly.

I wanted something I could drop in a folder and open directly from the filesystem — no account, no internet, no npm install. The whole point of the "W." prefix is that it's weighted — that's the useful part that most quick tools skip. Raw scores without weights are just vibes with extra steps.

Also I just wanted to write some vanilla JS for once without reaching for a framework. Turned out fine. The whole thing is ~200 lines of logic.

§3 Architecture / Design

Everything lives in a single index.html. The state is three plain JS arrays — crits, opts, and a flat scores object keyed by "optId|critId". Every mutation calls render() which rewrites the relevant DOM sections. No virtual DOM, no diffing, just targeted innerHTML rewrites on small panels.

The weighted score calculation runs in calculate(): it normalises criterion weights against their total, multiplies each score by the normalised weight, sums per option, and returns a sorted array. Results re-render on every stepper click and every weight change.

State & scoring core
javascript 
let crits = [], opts = [], scores = {};

// stable key for the scores lookup table
function sk(o, c) { return o + '|' + c }
function gs(o, c) { return scores[sk(o, c)] ?? 5 }
function ss(o, c, v) {
  scores[sk(o, c)] = Math.min(10, Math.max(0, v));
  renderResults();
}

// weighted score — normalise weights first so they don't need to sum to anything
function calculate() {
  const tw = crits.reduce((s, c) => s + c.weight, 0) || 1;
  return opts.map(o => {
    const raw = crits.reduce(
      (s, c) => s + gs(o.id, c.id) * (c.weight / tw), 0
    );
    return { id: o.id, name: o.name, score: +raw.toFixed(2) };
  }).sort((a, b) => b.score - a.score);
}
Stepper — per-cell score control
javascript 
// called by inline onclick="step(oid, cid, ±1)" from rendered table
function step(oid, cid, delta) {
  const next = Math.min(10, Math.max(0, gs(oid, cid) + delta));
  ss(oid, cid, next);
  document.getElementById(`sv-${oid}-${cid}`).textContent = next;
}

// weight slider — normalised on read so no re-render of the whole table
function setWeight(id, v) {
  const c = crits.find(c => c.id === id);
  if (c) {
    c.weight = +v;
    document.getElementById(`wv-${id}`).textContent = v;
  }
  renderResults();
}
Results renderer — bars animate via rAF
javascript 
function renderResults() {
  const el = document.getElementById('results');
  if (!opts.length || !crits.length) {
    el.innerHTML = '<p class="result-empty">—</p>';
    return;
  }
  const data = calculate();
  // render bars at width:0 first, then rAF to trigger CSS transition
  el.innerHTML = data.map((r, i) => `
    <div class="result-row">
      <span class="r-rank${i === 0 ? ' top' : ''}">${i + 1}</span>
      <span class="r-name">${esc(r.name)}</span>
      <div class="r-bar"><div class="r-fill" style="width:0"></div></div>
      ${i === 0 ? '<span class="r-badge">winner</span>' : ''}
      <span class="r-score${i === 0 ? ' top' : ''}">${r.score.toFixed(1)}</span>
    </div>
  `).join('');
  requestAnimationFrame(() => {
    document.querySelectorAll('.r-fill').forEach((b, i) => {
      b.style.width = (data[i].score / 10) * 100 + '%';
    });
  });
}

v1.1
v1.1 — Animated results & the rAF trick

v1.0 rendered results correctly but the bars just snapped to their final width — no animation. The fix sounds obvious: set width: 0 in the HTML, then update to the real width. Problem is the browser batches layout, so if you write style.width = '0' and then immediately style.width = '60%' in the same frame it never sees the zero and the transition never fires.

The fix is to write the innerHTML with bars at width:0, then schedule the actual widths in a requestAnimationFrame callback. That forces the browser to paint the zero-width state first, then the CSS transition picks up the change. Looks clean. Zero libraries required.

⚠ Known issue: When you delete a criterion while options are scored, the results panel flickers once before settling. Tracked and will be fixed in v1.2.
